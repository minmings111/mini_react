# D Worker Tasks

이 문서는 작업자 `D`가 이번 단계에서 맡아야 할 일을 빠르게 확인하기 위한 개인 작업 정리 문서입니다.
기준 문서는 `DECISIONS.md`와 `README.md`이며, 이 문서는 그 내용을 `D` 역할 기준으로 다시 정리한 것입니다.

## 1. 역할 요약

- 작업자 `D`의 목적:
  - `render` 흐름과 DOM prop 처리를 연결합니다.
  - 데모를 통해 팀과 사용자가 전체 동작을 이해할 수 있게 돕습니다.

- 작업자 `D`의 개발 목표:
  - `render`와 `dom-props`를 안정화합니다.
  - `examples/`에서 상태 변화와 diff 결과를 이해할 수 있게 보여줍니다.

## 2. 담당 파일 범위

이번 단계에서 `D`가 우선적으로 책임지는 범위:

- `src/render.js`
- `src/dom-props.js`
- `examples/`

공통 문서 중 필요 시 업데이트 가능:

- `README.md`

팀 합의가 생길 때만 업데이트:

- `DECISIONS.md`

## 3. 이번 단계 필수 작업

### 3-1. Render 흐름 점검

- `render`가 새 Virtual DOM을 계산하는 단계라는 개념이 코드에서 흔들리지 않는지 확인합니다.
- 이전 스냅샷과 비교할 준비 흐름이 자연스럽게 이어지는지 확인합니다.
- `commit` 개념과 섞이지 않도록 함수 책임을 분리해서 봅니다.

### 3-2. DOM Props 처리 점검

- `src/dom-props.js`가 속성 반영 책임만 가지는지 확인합니다.
- 이벤트, 일반 속성, 텍스트 관련 처리가 현재 범위 안에서 안정적으로 동작하는지 확인합니다.
- 데모 편의용 문자열이나 시각화 로직이 `src/` 안으로 들어오지 않게 주의합니다.

### 3-3. 데모 구성

- 데모 코드는 반드시 `examples/` 아래에 둡니다.
- 사용자가 상태 변화 흐름을 눈으로 볼 수 있게 구성합니다.
- 최소한 아래 값들이 화면에서 보여야 합니다.
  - `state`
  - `oldVdom`
  - `newVdom`
  - `diff 결과`

가능하면 아래도 함께 보여줍니다.

- patch 개수
- patch type 종류
- 실제 DOM 반영 결과

### 3-4. 설명 가능성 확보

- 데모를 보는 사람이 `render`와 `commit`의 차이를 이해할 수 있어야 합니다.
- 예제 화면의 문구와 패널 구조가 학습용 목적에 맞게 명확해야 합니다.
- 코어 엔진 코드와 학습용 UI 코드는 분리합니다.

## 4. 작업 시 주의사항

- `src/`에는 엔진 코드만 둡니다.
- 학습용 문구, 버튼 안내 문구, 시각화용 패널 UI는 `src/`에 넣지 않습니다.
- `MOVE` patch는 이번 단계 범위가 아니므로 다루지 않습니다.
- React와 동일한 수준의 최적화는 목표가 아닙니다.
- 다른 역할의 핵심 책임 파일은 함부로 수정하지 않습니다.
- 다른 역할 영역 수정이 필요하면 팀과 먼저 합의합니다.

## 6. 완료 기준

아래 항목이 만족되면 `D` 역할 작업이 이번 단계에서 어느 정도 완료된 것으로 볼 수 있습니다.

- `render` 흐름이 코드상에서 명확하다.
- DOM prop 처리가 현재 범위에서 안정적으로 동작한다.
- `examples/`에서 상태 변화 흐름을 눈으로 볼 수 있다.
- `oldVdom`, `newVdom`, `diff 결과`를 화면에서 확인할 수 있다.
- 가능하면 patch 개수와 patch type도 확인할 수 있다.
- 데모용 UI 코드가 `src/`가 아닌 `examples/`에 분리되어 있다.
